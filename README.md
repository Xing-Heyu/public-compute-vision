# public-compute-vision
An open-source manifesto for public computing resource sharing - 公共算力共享的开源宣言

## 🚫 明确边界（我们不做什么）

1. **不做商业平台**：不提供SLA，不保证可用性
2. **不做管理中心**：不设管理员，不仲裁纠纷
3. **不做激励机制**：无积分，无排名，无竞争
4. **不做完美系统**：承认所有未解决问题，但依然开始

## 🤝 开源治理模式

### 1. 无层级结构
- 只有"贡献者"，没有"维护者"
- 所有决策在Issues公开讨论
- 不设投票机制，靠共识推进

### 2. 异步协作
- 无强制会议，无定期同步
- 文档优先于代码，讨论优先于实现
- 贡献者按自己的节奏参与

### 3. 坦然面对停滞
- 如果项目无人参与，就让它安静
- 如果有人感兴趣，随时可以重启
- 代码永远在那里，思想永远开放

## 💎 为什么现在需要这个？

### AI时代的算力不平等正在加剧：
- **大公司**：拥有数据中心，算力近乎无限
- **中小企业**：租赁云服务，成本日益高昂
- **个人/散户**：个人设备有限，创意被硬件束缚

### 我们的应对：
不是建造另一个"算力发电厂"，而是编织一张"算力互助网"。  
当每个人贡献一点点闲置，每个人就能获得远超个体的能力。

项目白皮书：开源轻量化分布式算力/存储加速器 1. 一句话说清这是啥 一个开源的、非商业化的工具。它能把你的计算任务或大文件，自动调度到网络中其他人的闲置电脑上去处理或存储，完事了再把结果还给你。 2. 核心设计原则 •  开源与反商业化：所有代码永久开源，核心功能永不收费。  •  极简可行：先做出能跑通核心流程的最小版本，复杂功能靠社区后续补充。  •  用户掌控：你的私人文件不会被平台触碰或删除。  •  安全透明：所有操作有日志，代码可审查。   3. 方案调整与取舍建议【修改重点】 ▶ 关于技术可行性
原方案非常完整，但为了确保我们能先做出一个“能跑”的版本，建议明确开发优先级： •  核心先行：必须优先完成“网页提交 → 调度中心匹配 → 节点执行 → 返回结果”这条最核心的算力任务链路。这是项目的技术灵魂。  •  存储后置：分布式文件存储（分片、多节点存储）技术复杂度高，建议标记为 “二期功能” ，先确保算力调度这条腿能走稳。  •  简化网络：初期放弃复杂的P2P直传，采用 “客户端⇄服务器⇄用户” 的中转模式，以绕过家庭宽带无公网IP的难题。   ▶ 关于实用性演进【新增观点】
这个框架最大的潜力可能不是“存电影”，而是跑 “个人AI模型” 。 •  自然契合：训练或微调个人AI模型正是高算力需求场景。你的框架天生就是为“把计算任务打包送出去跑”而设计的。  •  叙事升级：这能让项目从“分布式脚本运行器”升级为 “去中心化个人AI算力网络”的雏形，格局和价值感完全不同。   4. 系统架构（简化版） 整个系统由三部分组成，分工明确： text 复制   下载    [用户] --> [网页前端] --> [调度中心] --> [节点客户端] --> [闲置电脑]  •  网页前端：你提交任务、看进度、下载结果的操作台。  •  调度中心：大脑。负责接收任务、找到闲置的节点、派活并跟踪状态。  •  节点客户端：安装在闲置电脑上的小程序。它负责上报自己是否空闲、领取任务、在严格隔离的“沙箱”中执行任务，并把结果回传。   5. 文件夹与安全设计【关键机制】 这是豆包方案里的精华设计，能有效保护提供算力的用户： text 复制   下载    在你的D盘（或你指定的位置）会生成：
